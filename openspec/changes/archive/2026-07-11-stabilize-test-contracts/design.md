## 背景

报告验收测试当前断言中文字面标题，但报告生成器输出英文 Markdown。另一方面，配置测试会继承 `config.local.yaml`，DashScope 包装脚本测试可能在到达既定缺失密钥路径前就因 Shell 配置发现而失败。因此，测试结果会随展示语言、开发者密钥和宿主 Shell 行为变化。

## 目标与非目标

**目标：**

- 让报告流程断言验证 Markdown 文档结构和报告数据，而不是标题语言。
- 在显式隔离的配置上下文中执行缺失密钥检查。
- 在支持的 Shell 上保留配置诊断和 DashScope 包装脚本面向用户的既定退出码。

**非目标：**

- 不改变报告渲染语言、报告数据模型或下载元数据。
- 不移除或转换条件跳过。
- 不在单元测试中要求真实 DashScope 凭据。

## 决策

### 断言报告语义而非本地化标签

验收测试将检查标题和区段的层级与顺序、必需的来源/问题/摘要/时间线值、事件格式和可选校验反馈内容，不比较本地化标题文本。这保留了报告契约覆盖，同时允许展示层本地化。

备选方案是统一所有报告标题语言。该方案被拒绝，因为输出语言明确不是契约，且会无谓改变用户可见输出。

### 让测试配置输入显式化

验证缺失 API Key 的测试将移除相关环境变量，并阻止 `config.local.yaml` 参与配置解析。完整配置状态由测试夹具或调用边界提供，而不是由开发者机器决定。

备选方案是将本地密钥视为测试中的有效配置并更新成功预期。该方案被拒绝，因为测试的目的正是验证缺失密钥诊断，在存在本地密钥的机器上会失去覆盖。

### 在 Shell 专属解析前验证包装脚本前置条件

DashScope 包装脚本入口将在可选 Shell/配置解析前使缺失密钥判断可达，该解析在 PowerShell、Git Bash 和 Linux 间可能不同。既定缺失密钥退出码保持稳定；其他真实解析错误仍保留各自的错误行为。

备选方案是放宽包装脚本测试以接受任意非零退出码。该方案被拒绝，因为会移除调用方依赖的明确错误契约。

## 风险与取舍

- [结构化断言可能遗漏意外的标题文本回归] -> 若确有稳定措辞需求，单元测试可保留聚焦生成器的检查；验收流程保持语言无关。
- [配置隔离不同于生产发现流程] -> 仅诊断测试隔离输入；生产优先级和 `config.local.yaml` 支持不变。
- [提前验证包装脚本会遮蔽配置解析错误] -> 仅限无密钥路径；提供密钥时保留既有行为。

## 迁移计划

1. 在隔离的开发分支上更新受影响测试，并在需要时更新包装脚本行为。
2. 使用工作区本地临时目录运行聚焦的报告、配置和 DashScope 测试组。
3. 在同一临时目录配置下运行全量测试，并保留现有条件跳过。
4. 不需要数据迁移或运行时部署迁移。
