## Context

当前 Ruff 配置选择 `E/F/I/N/W/UP`，但使用了已弃用的顶层 `select` 配置。`src/` 有 124 个静态问题，主要包括 67 个导入排序、23 个长行、11 个引号类型注解、10 个旧集合导入、7 个未使用导入和 1 个未使用局部变量；63 个源码文件不满足 Ruff 格式。部分 `F401` 出现在依赖导入副作用完成注册的模块，不能无条件自动删除。

## Goals / Non-Goals

**Goals:**

- 让 `src/` 和 `tests/` 满足统一 Ruff 检查与格式门禁。
- 清除明确的死导入、死变量和过时语法。
- 保持工具/代理注册副作用及运行时行为不变。

**Non-Goals:**

- 不在本 change 中拆分长模块或重构业务流程。
- 不扩大 Ruff 规则集合，不引入新的格式化或类型检查依赖。
- 不检查 `frontend/original-ui`。

## Decisions

1. 先修复 `stabilize-test-contracts`，保证格式和静态修改之后有可信的全量测试门禁。
2. 将 Ruff 规则迁移到 `[tool.ruff.lint]`，保留现有规则集合与 120 列限制，避免配置迁移同时扩大政策。
3. 分两类处理问题：导入排序、类型现代化和格式使用 Ruff 的安全修复；`F401`、`F841`、长字符串和注册模块逐项人工确认。
4. 注册入口使用显式别名、`__all__` 或说明性 `noqa` 表达副作用，不删除实际承担注册职责的导入。
5. 静态检查、格式检查和全量测试作为同一验收门禁，任何行为回归都优先回滚对应机械修改。

## Risks / Trade-offs

- [风险] 自动删除导入可能破坏注册副作用。 -> 对注册模块禁用批量删除并增加注册表回归测试。
- [风险] 全仓格式化产生较大差异。 -> 单独提交机械格式化，并与语义修复分开审查。
- [风险] 长行可能是稳定 prompt 文本。 -> 对字符串采用结构化拼接或规则豁免，不改变内容字节。

## Migration Plan

1. 记录 Ruff JSON 基线并迁移配置位置。
2. 应用安全自动修复和格式化。
3. 人工处理剩余问题，运行针对性测试和全量测试。
4. 若行为验证失败，按类别回滚而不是关闭规则。
